## 1. Executive Summary

Algo Safe is a single-file, policy-driven M-of-N smart account for Algorand (`contract.algo.ts`, 1,514 lines — up from 1,432 at the last audit, reflecting the five fixes and their inline documentation). This audit reviewed the commit that actually shipped the prior audit's five recommended fixes to `main`, and independently re-derived — rather than assumed — that each fix closes the issue it targets:

- **H-01** (multi-chunk payload bait-and-switch): closed by `Proposal.payloadVersion`, bumped on every `appendTransactionGroupPayload` write and required as an argument to `approveProposal` (`contract.algo.ts:665-681`). Verified against the regression test at `contract.e2e.spec.ts:2524-2614`, which reproduces the exact front-running race and confirms the stale approval reverts.
- **M-01** (`paused` unreachable): closed by the new `ADM_SET_PAUSED` admin-change type (10), correctly scoped to gate only fund-moving transaction-group creation/append/execution while leaving all governance — including the unpause itself — reachable (`contract.algo.ts:550,594,869`). Verified against `contract.e2e.spec.ts:2616-2681`.
- **M-02** (no aggregate chunk-length bound): closed by `Proposal.totalTxns`, correctly maintained across overwrites and bounded to `MAX_GROUP_TXNS` (16) on every append (`contract.algo.ts:605-611`). Verified against `contract.e2e.spec.ts:2683-2737`, including the exact-boundary (16 succeeds, 17 rejected) and overwrite-non-inflation cases.
- **M-03** (bootstrap-path mixing desyncs `activePrivGroupCount`): closed by an explicit `groupCount === 0` guard in `bootstrap()` (`contract.algo.ts:394`), making the two bootstrap paths mutually exclusive by assertion rather than silently corruptible. Verified against `contract.e2e.spec.ts:2739-2758`.
- **L-01** (no defensive self-`appId` check): closed by an explicit assert in `_validateApp` (`contract.algo.ts:1199`).

No **Critical** or **High** findings were identified in this fresh pass. The core invariants re-verified independently in this audit — proposal state-machine transitions, access-control re-checks at execution time, spend-limit close-out accounting, the governance-lockout guard, and box-key collision-freedom — remain sound, and the five fixes genuinely close the gaps the prior report identified without introducing new gaps of comparable severity.

This audit did identify one **Medium** finding: `CLAUDE.md` and `PRODUCT-DESCRIPTION.md` were not updated alongside the v1.7.0 changes, and now actively misdocument the contract — `PRODUCT-DESCRIPTION.md:611` still shows `approveProposal(proposalId, budget)`, the pre-fix two-argument signature, when the deployed contract requires a third, non-optional `expectedPayloadVersion` argument; `PRODUCT-DESCRIPTION.md:453` still describes `paused` as "(reserved)" (implying it has no effect) when `ADM_SET_PAUSED` now makes it fully functional; and `CLAUDE.md`'s "Breaking changes seen across versions" section — the document explicitly designated as the source of truth for ABI breaks — has no entry for v1.6.0/v1.7.0 at all. Two **Low** findings and three **Informational** items round out the report, covering a state-machine consistency gap (`appendTransactionGroupPayload` doesn't check proposal expiry), a client-library correctness gap in the off-chain keyreg mapping, and process/testing observations.

**Overall assessment**: the contract is materially more robust than the version audited previously, and the fix implementations show evidence of careful engineering (e.g. the M-01 fix's deliberate avoidance of a self-lockout that a naive "block everything while paused" implementation would have introduced — see the code comment at `contract.algo.ts:865-868`). The one Medium finding is a documentation-only gap with no on-chain security impact, but should be closed promptly since it currently misdirects integrators about a required-argument ABI break.

**Update**: all three open findings (M-01, L-01, L-02) were fixed the same day — the L-01 contract change shipped as `contract.algo.ts` v1.8.0 (approval hash `d66a4b63...7d10`), with 2 new regression tests bringing the suite to 65/65 passing. See the "Remediation Update" section above and the "Remediation" note under each finding. These fixes are in the working tree and have not yet been committed.

## 4. Verification of Prior Findings (H-01, M-01, M-02, M-03, L-01)

This section documents the independent re-derivation referenced in §2. For each finding: what the code now does, the specific lines, and the regression test that exercises it.

### H-01 (prior report) — Multi-Chunk Proposal Bait-and-Switch — **Confirmed Fixed**

`Proposal.payloadVersion` (`contract.algo.ts:184-188`) initializes to `1` in `_newProposal` (`contract.algo.ts:916`) and increments on every successful `appendTransactionGroupPayload` write (`contract.algo.ts:617`). `approveProposal` takes a required `expectedPayloadVersion: uint64` argument and asserts it equals the proposal's live `payloadVersion` before recording the approval (`contract.algo.ts:665,678`: `assert(proposal.payloadVersion === expectedPayloadVersion, 'payload changed since review')`). A proposer editing chunk content after an approver has decided to approve, but before that approval lands, now bumps the version and causes the pending approval to revert instead of silently binding to the new content.

Independently traced against `contract.e2e.spec.ts:2524-2614` ("rejects a stale approval when a multi-chunk payload is edited after review but before confirmation (H-01 regression)"): the test creates a 2-chunk proposal, records `payloadVersion` after an honest edit, has the proposer rewrite chunk 2 to redirect funds to an attacker address, and asserts the approver's `approveProposal` call (bound to the stale version) throws — confirmed the assertion is exercised, not merely present in dead code.

### M-01 (prior report) — `paused` Unreachable / Inconsistent — **Confirmed Fixed**

`ADM_SET_PAUSED` (changeType `10`, `contract.algo.ts:105-110`) is gated by the default (`PRIV_GROUP`) branch of `_assertPrivilegeForChange` (`contract.algo.ts:1290-1296`) and reuses `AdminChange.activeFlag` as the desired paused state. The `paused` check was correctly scoped to **only** the fund-moving path: `proposeTransactionGroup` (`contract.algo.ts:550`), `appendTransactionGroupPayload` (`contract.algo.ts:594`), and the `PT_TRANSACTION_GROUP` branch specifically inside `_executeProposalInternal` (`contract.algo.ts:869`) — never `proposeAdminChange`, `approveProposal`, or the `PT_ADMIN` execution branch. This means an `ADM_SET_PAUSED`-type unpause proposal can always be proposed, approved, and executed even while the safe is paused, avoiding the self-lockout a naive "block everything" implementation would introduce (explicitly called out in the code comment at `contract.algo.ts:865-868`).

Independently traced against `contract.e2e.spec.ts:2616-2681`: confirms new transaction-group proposals, appends, and execution are all blocked while paused, then confirms `ADM_SET_PAUSED` (to unpause) succeeds while still paused, and the previously blocked proposal executes immediately after.

### M-02 (prior report) — No Aggregate Transaction-Count Bound — **Confirmed Fixed**

`Proposal.totalTxns` (`contract.algo.ts:189-193`) is set from `payload.length` in `proposeTransactionGroup` (`contract.algo.ts:557`) and maintained in `appendTransactionGroupPayload`, which computes the new aggregate by subtracting any existing stored chunk's length before adding the new one (correctly handling in-place overwrites of an already-written slot — `contract.algo.ts:605-611`) and asserts `newTotal <= MAX_GROUP_TXNS` (16) before storing.

Independently traced against `contract.e2e.spec.ts:2683-2737`: builds a proposal across all 6 slots reaching exactly 16 total transactions (succeeds), re-edits the last slot with an equal-sized replacement (succeeds, confirming overwrites don't double-count), then attempts to grow it to 17 (rejected). This is a genuine boundary test, not just a "large input rejected" smoke test.

### M-03 (prior report) — Bootstrap-Path Mixing Desyncs `activePrivGroupCount` — **Confirmed Fixed**

`bootstrap()` now asserts `this.groupCount.value === Uint64(0)` (`contract.algo.ts:394`, `'bootstrapGroup already used; cannot mix bootstrap paths'`) before touching `activePrivGroupCount`. Combined with the pre-existing `bootstrapped === 0` guard on both paths (`bootstrap()` sets `bootstrapped = 1` at the end; `bootstrapGroup()` and `finalizeBootstrap()` both require `bootstrapped === 0`), the two bootstrap paths are now fully mutually exclusive in both call orders: `bootstrapGroup()` → `bootstrap()` is blocked by the new `groupCount === 0` assert, and `bootstrap()` → `bootstrapGroup()` was already blocked by the `bootstrapped` flag.

Independently traced against `contract.e2e.spec.ts:2739-2758`: calls `bootstrapGroup` once, then asserts a subsequent `bootstrap()` call throws.

### L-01 (prior report) — No Defensive Self-`appId` Check — **Confirmed Fixed**

`_validateApp` now asserts `tx.appId !== Global.currentApplicationId.id` (`contract.algo.ts:1199`, `'self-call not allowed'`) before staging an app-call transaction-group entry, giving a clear contract-level error instead of relying solely on the AVM's protocol-level self-call rejection.

### L-01 (this report): `appendTransactionGroupPayload` Does Not Check Proposal Expiry

**Severity**: Low
**Status**: Open
**Component**: AlgoSafe (`contract.algo.ts`)
**File**: `contract.algo.ts:582-622`

**Description**:
Every other state-mutating proposal method checks `Global.round` against `proposal.expiryRound` before acting: `_newProposal` requires `expiryRound > Global.round` at creation (`contract.algo.ts:902`), `approveProposal` requires `Global.round <= proposal.expiryRound` (`contract.algo.ts:671`), and `_executeProposalInternal` requires the same (`contract.algo.ts:848`). `appendTransactionGroupPayload` checks `proposal.status === STATUS_ACTIVE || STATUS_READY` (`contract.algo.ts:599`) but never checks `proposal.expiryRound` against `Global.round`.

**Impact**:
No fund-loss or authorization-bypass path exists: an appended chunk on an already-expired proposal can never subsequently be approved or executed, since both of those paths independently re-check expiry. The practical effect is limited to a proposer being able to pay fees and consume box-write budget editing a proposal that can never do anything — pure wasted effort, not a security gap. It is included here as a Low/consistency finding because it breaks the otherwise-uniform pattern the rest of the state machine follows (every other mutator checks expiry), which is exactly the kind of small inconsistency the checklist in `AI-AUDIT-INSTRUCTIONS.md` ("A proposal cannot be approved after expiry... or after it is no longer ACTIVE/READY") is designed to catch, and a future refactor that relies on `appendTransactionGroupPayload`'s status check as a stand-in for "is this proposal still alive" would be relying on an incomplete proxy.

**Proof of Concept**: N/A (no reachable harm; a static-analysis-style consistency observation, not an executable exploit).

**Recommendation**: Add `assert(Global.round <= proposal.expiryRound, 'proposal expired')` to `appendTransactionGroupPayload`, matching the pattern in `approveProposal`/`_executeProposalInternal`. This is a pure hardening change (rejects a call that already can never lead anywhere) with no behavioral downside, and would be a good candidate to bundle with the next version bump rather than justify one on its own.

**Remediation (2026-07-07)**: Implemented as recommended — the assert was added to `appendTransactionGroupPayload` immediately after the status check, and `CONTRACT_VERSION` was bumped to `v1.8.0` (this ended up being the change that justified the bump rather than piggy-backing on another). Regression test added: `'rejects appending a payload chunk to an expired proposal (2026-07-07-v2 L-01 regression)'` in `contract.e2e.spec.ts`, which creates a short-expiry proposal, advances the chain past its expiry round, and asserts the append reverts. Status: fixed in the working tree, not yet committed.

### L-02 (this report): Off-Chain `algosdkTxnToSafeTxn` Misclassifies a Standard "Go Offline" Key Registration as Online

**Severity**: Low
**Status**: Open
**Component**: `algo-safe` npm package (`src/safe-tx.ts`)
**File**: `projects/algo-safe-contracts/src/safe-tx.ts:391-402`

**Description**:
`algosdkTxnToSafeTxn`'s keyreg branch derives the contract's `online` flag as:

```ts
online: kr?.nonParticipation ? 0n : 1n,
```

On Algorand, the conventional way to submit a "go offline" key-registration transaction is to omit the vote/selection/state-proof keys (leave them `undefined`) while leaving `nonParticipation` `false`/unset — `nonParticipation: true` is a distinct, stronger, and effectively irreversible "opt out of consensus permanently" flag, not the normal offline transition. A caller building a standard offline-registration transaction via algosdk's usual helpers and converting it through `algosdkTxnsToSafeTxnGroup` would have `kr.nonParticipation` be falsy, so this mapping sets `online: 1n` — the *opposite* of the caller's intent — while `voteKey`/`selectionKey`/`stateProofKey` all fall through to `EMPTY_BYTES` (`contract.algo.ts:395-397`).

**Impact**:
On-chain, `_stageKeyReg` only sets the vote/selection/state-proof-key fields when `tx.online !== Uint64(0)` (`contract.algo.ts:1138-1147`); with `online = 1` and empty-byte keys, the resulting inner transaction would carry vote/selection keys of the wrong size, which the AVM is expected to reject at `itxn_submit` (these fields have fixed 32/32/64-byte size requirements). The most likely outcome is therefore a loud execution failure — a proposal that never successfully executes — rather than a silent misconfiguration that puts the safe online with garbage keys. This bounds the severity to Low (no fund-loss or safety path identified), but it is a genuine correctness gap in the client library: a caller relying on `algosdkTxnsToSafeTxnGroup` for an ordinary offline-registration transaction gets a proposal that will fail at execution for a non-obvious reason, after already having collected the group's approvals.

**Proof of Concept**: N/A (traced from the algosdk `KeyRegistrationTransactionFields` semantics and the `_stageKeyReg` size-enforcement path; not independently reproduced against LocalNet in this pass — see the corresponding item in §6).

**Recommendation**: Derive `online` from whether `kr?.voteKey` (or any of the three key fields) is present, e.g. `online: kr?.voteKey ? 1n : 0n`, rather than solely from `nonParticipation`. Add a unit test in `safe-tx.spec.ts`-equivalent coverage (or a small `on-chain.e2e.spec.ts` addition) round-tripping a keys-omitted, `nonParticipation`-unset keyreg transaction through `algosdkTxnsToSafeTxnGroup` and asserting `online === 0n`.

**Remediation (2026-07-07)**: Implemented as recommended — `online` is now `kr?.voteKey && !kr.nonParticipation ? 1n : 0n` (`src/safe-tx.ts`), covering the standard keys-omitted offline case, the explicit `nonParticipation: true` opt-out, and the keys-supplied online case. Regression test added: `'maps a standard keys-omitted "go offline" keyreg to online=0 (2026-07-07-v2 L-02 regression)'` in `contract.e2e.spec.ts`, exercising all three cases through `algosdkTxnsToSafeTxnGroup` → `decodeKeyRegTxn`. Status: fixed in the working tree, not yet committed.

### I-02: Missing Regression Tests for Three Checklist Scenarios

**Severity**: Informational
**Component**: Test coverage (`contract.e2e.spec.ts`)

Three scenarios explicitly called out in `AI-AUDIT-INSTRUCTIONS.md`'s checklists remain untested by the current 55-test suite (none indicate a suspected defect — the underlying code paths were independently read and appear correct; these are coverage gaps, not known bugs):

1. **Malformed/adversarial ARC4 payload bytes that don't match the declared `txType`.** No test stores a `SafeTxn` with, e.g., `txType = TX_PAYMENT` but truncated or wrong-shape `data`. `decodeArc4<T>` is expected to revert on a shape mismatch (this is how ARC4's self-describing tuple encoding is documented to behave), but this has not been exercised against the actual compiled contract. This is the same gap `RISK-REGISTRY.md` already tracks as R-16 ("Monitoring", unchanged by this audit).
2. **Removing the last member of a group entirely.** The existing `'removes a member through governance'` test (`contract.e2e.spec.ts:983-1022`) removes one of two members from a 2-member, 1-of-2 group; it does not exercise the boundary case of removing a group's sole remaining member (which `_adminRemoveMember`'s `assert(group.memberCount - 1 >= group.threshold, ...)` should reject for any `threshold >= 1`, but this specific boundary — distinct from the already-tested privilege-lockout scenarios — has no dedicated regression test).
3. **Proposal-expiry boundary for approval/execution, not just creation.** `'rejects proposals whose expiry is not in the future'` (`contract.e2e.spec.ts:1042-1054`) only tests the creation-time boundary (`_newProposal`'s `expiryRound > Global.round`). No test exercises `approveProposal`/`executeProposal`'s `Global.round <= proposal.expiryRound` check at the exact boundary (`Global.round === expiryRound` should still succeed; the following round should fail).

**Recommendation**: Add regression tests for all three; each is a small, mechanical addition to the existing suite's patterns (the M-02 regression test already demonstrates the project's boundary-testing style well).
